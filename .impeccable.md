# Design Context

## Users
**Who they are**: Developers and engineers who frequently use Git commands in their daily workflow. They range from junior developers to senior engineers who value efficiency and want to avoid typing long Git commands.

**Context**: Command-line terminal environment. Users are often in flow state, moving fast between tasks. They need quick, glanceable information with minimal friction.

**Job to be done**: Quickly navigate Git operations (branch switching, file staging, commit message generation) without memorizing complex command syntax or switches.

## Brand Personality
**Three words**: Simple Fast Powerful

**Voice & Tone**: Minimal friction, maximum productivity. No-nonsense efficiency wrapped in refined elegance. Every pixel and animation serves a purpose.

**Reference aesthetics**: Modern CLI tools like `eza`, `bat`, `ripgrep` - they feel fast, smart, and delightful without being flashy.

**Anti-references**: Bloated UIs with excessive animations, rainbow color schemes, overstyled terminal apps that prioritize looks over speed.

## Aesthetic Direction
**Visual tone**: Clean, focused, professional. Dark-terminal-native with warm, nuanced color palettes.

**Theme**: Leverages 5 curated color themes (Catppuccin Mocha, Rose Pine, Tokyo Night, Kanagawa, Dracula) - each provides warm, sophisticated alternatives to harsh terminal defaults.

**Key principles**:
- **Performance over decoration**: Every visual choice must enhance usability, not slow it down
- **Semantic meaning over decoration**: Color should communicate state, hierarchy, and context - not just look pretty
- **Refined minimalism**: Elegant spacing, thoughtful typography, subtle hierarchy - never cluttered or overwhelming

## Design Principles

1. **Purposeful Color**: Every color must enhance understanding - status, state, hierarchy, or context. No decoration for decoration's sake.

2. **Terminal-Native**: Work *with* the terminal environment, not against it. Respect theme choices, avoid harsh contrasts, embrace the medium.

3. **Glanceability First**: Users should understand the state in under 100ms. Clear visual hierarchy, status indicators, and semantic coloring.

4. **Warm over Cold**: Prefer warm color temperatures (pinks, oranges, warm neutrals) over harsh blues and greens when appropriate. But maintain semantic clarity.

5. **Contextual Richness**: The same element can change appearance based on Git state - current branch, modified files, branch status. Adapt intelligently.

## Color Usage Guidelines

### Semantic Color Mapping
- **Current/Active**: Use `highlight` color (often purple/pink) to make the current selection unmistakable
- **Modified/Changed**: Use `warning` color (orange/yellow) to draw attention to modifications
- **Success/Clean**: Use `success` color (green) to indicate healthy state
- **Error/Conflict**: Use `error` color (red) to highlight problems
- **Neutral/Default**: Use `text` or `muted` colors for general content

### State-Based Coloring
- Don't make all branches the same color - differentiate by Git state
- Current branch: `highlight` color + indicator icon (★)
- Branch with changes: `warning` color
- Clean branch: default `primary` color
- Remote branch: `muted` color

### Visual Hierarchy
- **Primary actions**: Bold, high contrast
- **Secondary information**: Muted, smaller
- **Tertiary/context**: Subtle, minimal ink
- **Groupings**: Use color to separate logical sections, not just borders

## Accessibility
- Maintain WCAG contrast ratios for all text on backgrounds
- Color should *support* meaning, never be the sole indicator
- Combine color with icons, labels, or position for critical information
- Support colorblind users through redundant encoding (shape + color)
